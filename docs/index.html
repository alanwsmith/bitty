bitty.js

tiny component. big reactivity.

Hello, bitty

bitty is a web component. It provides reactivity to the elements it surrounds. For example, this "Click me" button updates the text with a timestamp.

No clicks yet

bitty uses a signaling approach. data-* attributes define the signals elements send and receive. Here's the HTML: 

<bitty-js data-bridge="./hello-text.js">
  <button data-send="update">Click me</button>
  <div data-receive="update">No clicks yet</div>
</bitty-js>

and the corresponding hello-text.js JavaScript:

export default class {
  update(el, _event) {
    el.innerHTML = `Clicked at ${Date.now()}`;
  }
}

How It Works

  • The data-bridge attribute on the bitty-js element makes the connection to the JavaScript module responsible for providing functionality. hello-text.js in this case.
  • The data-receive attribute on the div registers it to receive updates when the updateText signal is sent.
  • The data-send attribute on the button sends the updateText signal when the button is clicked.

Full Attribute List

The bitty API is comprised of Six (TODO: maybe just four) attributes. They are:

  • data-bridge - The path to the JavaScript module responsible for providing functionality for the bitty component and its elements.

    Only used by the bitty-js element itself.

  • TODO: See about deprecating data-call in favor of just using data-send without corresponding data-receive since the events are passed the same way.

    data-call - Calls a function without sending a signal. These calls are made prior to any signals being sent.

    Can be used in either the bitty-js element itself or any child elements.

  • data-ignore - TKTKTKTK (in line for deprication)

    TKTKTKT: Usage location(s)

  • data-listeners - Adds additional event listeners to the component. (The defaults are click, input, and change.)

    Only used by the bitty-js element itself.

  • data-receive - Sets up child elements to receive signals. The value is the name of the funciton that's used for the signal.

    Only used by the child element.

  • data-send - Defines signals for the element to send. The value is the name of the funciton that's used for the signal.

    Signals are sent from the bitty-js element when the component is initialized. They are sent from child elements when one of their event listeners fires.

    Can be used in either the bitty-js element itself or any child elements.

bitty is really just wiring between elements and functions in the module called by data-bridge. The Examples below provide more documentation and details on how bitty works.

Storing State

waiting

</bitty-js>

export default class {
  #counter = 0;

  displayCount(el, _event) {

    el.innerHTML = this.#counter;
  }
}

  • Basic data-bridge, data-send, anddata-receive example.

  • The initial value is set inside the class as a private element on initilization. It is updated directly when the displayCount signal is sent.

Calling Functions


<bitty-js data-bridge="./calling-functions.js"
    data-send="displayCount">
    <button
        data-call="increment"
        data-send="displayCount">
        Click Me
    </button>
    <div data-receive="displayCount"></div>
</bitty-js>

export default class {
  #counter = 0;

  increment(_event) {
    this.#counter += 1;
  }

  displayCount(el, _event) {
    el.innerHTML = this.#counter;
  }
}

  • The initial value is set as and stored in a #counter private element on initilization.

  • data-send is used on the bitty-js element to send the displayCount signal which is picked up by the div to show the initial #counter value when the component is initialize.

  • The button uses data-call to trigger the increment function that updates the #counter variable before any signals are sent.

  • data-send on the button triggers the displayCount signal and corresponding function when the button is clicked.

Additional Data Attributes

The quick brown fox

<bitty-js data-bridge="./additional-attributes.js">
  <div>
    <button
        data-send="setClass"
        data-color="red">Red</button>
    <button
        data-send="setClass"
        data-color="green">Green</button>
    <button
        data-send="setClass"
        data-color="blue">Blue</button>
  </div>
  <div data-receive="setClass">The quick brown fox</div>
</bitty-js>

export default class {
  #colors = [
    "blue",
    "green",
    "red",
  ];

  setClass(el, event) {
    this.#colors.forEach((color) => {
      if (color === event.target.dataset.color) {
        el.classList.add(color);
      } else {
        el.classList.remove(color);
      }
    });
  }
}

  • Additional data-* attributes can be used to send supplemental data through signals.

  • The attributes are available as normal in the event that's passed as the second argument to the function.

Updating Range Sliders


<bitty-js data-bridge="./updating-range-sliders.js">
  <div>
    <label for="update-values-input-slider">Input Slider</label>
    <input
      type="range"
      id="update-values-input-slider"
      data-send="sliderValue">
  </div>
  <div>
    <label for="update-values-output-slider">Output Slider</label>
    <input
      type="range"
      id="update-values-output-slider"
      data-receive="sliderValue">
  </div>
</bitty-js>

export default class {
  sliderValue(el, event) {
    el.value = event.target.value;
  }
}
  • This is a one way update between sliders. The output slider gets updated with the value of the input slider. But, the input slider is not effected by changes to the output slider.

Color Picker


  <style>
    .color-picker {
      background-color: var(--color-picker-background);
      color: var(--color-picker-text);
    }
  </style>
  <bitty-js
      data-call="initMode"
      data-batch="sliders"
      data-bridge="./color-picker.js"
      class="color-picker">
    <div>
      <label for="slider-l">Lightness</label>
      <input
        type="range"
        min="0"
        max="100"
        step="0.01"
        id="slider-l"
        data-call="handleSlider"
        data-receive="lValue"
        data-key="l"
      />
      <label for="slider-c">Chroma</label>
      <input
        type="range"
        min="0"
        max="0.3"
        step="0.00001"
        id="slider-c"
        data-call="handleSlider"
        data-receive="cValue"
        data-key="c"
      />
      <label for="slider-h">Hue</label>
      <input
        type="range"
        min="0"
        max="360"
        step="0.01"
        id="slider-h"
        data-call="handleSlider"
        data-receive="hValue"
        data-key="h"
      />
    </div>
  </bitty-js>

export default class {

  #batches = {
    "sliders": [
      "cValue",
      "hValue",
      "lValue",
    ],
  };

  #data = {
    "activeMode": "light",
    "modes": {
      "light": {
        "l": 70,
        "c": 0.05,
        "h": 190,
      },
      "dark": {
        "l": 20,
        "c": 0.1,
        "h": 40,
      },
    },
  };

  get batches() {
    return this.#batches;
  }

  handleMode(data) {
    this.#data.activeMode = data.target.value;
    this.updateStyles();
  }

  handleSlider(data) {
    const modeKey = this.#data.activeMode;
    const key = data.target.dataset.key;
    this.#data.modes[modeKey][key] = data.target.value;
    this.updateStyles();
  }

  initMode() {
    this.updateStyles();
  }

  cValue(el, _event) {
    el.value = this.#data.modes[this.#data.activeMode].c;
  }

  hValue(el, _event) {
    el.value = this.#data.modes[this.#data.activeMode].h;
  }

  lValue(el, _event) {
    el.value = this.#data.modes[this.#data.activeMode].l;
  }

  backgroundColorVar() {
    const mode = this.#data.modes[this.#data.activeMode];
    return `oklch(${mode.l}% ${mode.c} ${mode.h})`;
  }

  textColorVar() {
    const modeKey = this.#data.activeMode === "light" ? "dark" : "light";
    let mode = this.#data.modes[modeKey];
    return `oklch(${mode.l}% ${mode.c} ${mode.h})`;
  }

  template() {
    return `

    `;
  }

  updateStyles() {
    this.bridge.style.setProperty(
      `--color-picker-background`,
      this.backgroundColorVar(),
    );
    this.bridge.style.setProperty(
      `--color-picker-text`,
      this.textColorVar(),
    );
  }

}
Release Notes

  • Version: 0.3.0

    DATE TBD

    Data attribute naming update for clarity

    • The biggest update is changing data-b, data-c, data-r, and data-s to data-batch, data-call, data-receive, and data-send, respectivally. I originally use the shorter names to reduce the text length. The increased clarity of the longer names is worth the few extra characters.
    • The other big change is removing the leading _ and $ characters from function names. They were originally put in place to create a naming convention that differentiated between functions that were hit with data-call and data-send. In practice, that wasn't necessary.
    • Removed the data-batch attribute. While there are some use cases where it might be nice, the extra complexity, mental overhead, and maintance aren't worth it.
    • Added documentation to each of the examples. (Documention is stored as markdown direclty in the example files.)
    • Split the web site page up into individual template for sections. Much nicer to work with.
    • Updated example functions to always use (el, _event) instead of (el, _) for clarity.
    • TODO: Confrim inert/include/ignore can be dropped from the top level components then remove them.
    • TODO: Confirm async/await stuff can be done directly with functions and there doesn't need to be anything specific added to the element signals to acommodate them (i.e. elements shouldn't have to know or make a decision about if something needs to be called async)

  • Version: 0.2.3

    June 5, 2025

  • Version: 0.1.0

    June 2, 2025

    Getting the project started.

    • Initial prototype.
    • <bitty-js> wraps elements to provide them with reactive capabilities.
    • Basic data-c, data-s, and data-r in place to call functions, send, and receive updates, respectively.
    • Functionality is loaded using data-wrapper to point to a module to load. The module must export a Wrapper class that gets loaded and used to provide functions and send/receive callbacks.
    • Uses string prefixes to make determinations about how to handle data through the send/receive channels (e.g. htmlSOMETHING updates .innerHTML of an element while valueSOMETHING updates the .value).
    • Defined convention for functions. data-c maps to functions in the Wrapper class that start with a _ (underscore). The data-s and data-r attributes map to functions in the Wrapper class that start with a $ (dollar sign).
    • Decided against using data-c data-s and data-r on the bitty.js tags. That would involved a lot of extra overhead in parsing to differentiate between the top level element and the content it wraps. Using data-send instead as a replacement for data-c. Others, TBD.
    • Set up data-c="batchSOMETHIGN" functionality to send a single single that gets turned into multiple signals in the Wrapper.
    • Defined convention of using .batches to look for batches. It must be a hash where the keys match the incoming request and the value is an array of functions to run.
    • Defined .bridge to allow Wrapper functions to access the parent bitty-js element.
    • Scoped event listeners to the bitty-js elements.
    • Set up data-listeners attribute on bitty-js tags to override the default listeners (which are input and click).
    • Created example that stores its own state.
    • Created example that updates an element with the same content it sent (i.e. verified feedback can be avoided).
    • Created example using data-send to load initial values.
    • Created example that sends multiple signals.
    • Created example with multiple elements receiving the same signal.
    • Created example showing how to update global CSS variables/properties.
    • Created example showing custom event listeners.
    • Created example showing how to update CSS variables/properties scoped to the content of individual bitty-js tags.
    • Examples use parent page's CSS for styling. It confirms I'm happy with the choice to use the light DOM instead of the shadow DOM.
    • Set up initial preflight check to ensure functions are in place.
